# Offline Replay Architecture

This document describes the current offline/PWA runtime as it exists in the repo on 2026-03-20.

It is intentionally narrower than the general PWA setup guide in `docs/setup/PWA.md`.

## Goals

The current offline lane is optimized for:

- truthful behavior
- low-conflict implementation ownership
- local diagnosability
- incremental extension of replay support

It is not yet a full offline-first product architecture.

## Current Source Of Truth

There is one offline mutation queue in the app today:

- local IndexedDB, managed by `src/lib/offline.ts`

There is no active server-side offline queue anymore.

What this means:

- Settings reflects local queue state only
- retry counts live in the client queue processor
- replay success/failure is determined client-side
- support/debugging does not currently have a durable server-side audit trail for failed offline actions

## Service Worker Ownership

The production app currently registers:

- `/service-worker.js`

That file is shipped from:

- `public/service-worker.js`

The repo also still emits:

- `/sw.js`

That worker is generated by `vite-plugin-pwa` during build, but the current source configuration disables automatic registration for it, and built HTML no longer injects plugin registration code.

Current practical ownership:

- app code owns registration through `src/lib/serviceWorker.ts`
- `public/service-worker.js` owns the effective runtime worker path
- `vite-plugin-pwa` still owns generated precache artifacts and emits `sw.js`, but that worker is not the registered runtime path today

This split is the main remaining PWA build-pipeline ambiguity.

## Queue Shape

Queued mutations are stored in IndexedDB with these effective fields:

- `mutationType`
- `mutationArgs`
- `status`
- `attempts`
- `timestamp`
- `syncedAt`
- `error`

Status meanings:

- `pending`: waiting to be replayed
- `syncing`: actively being replayed
- `synced`: replay succeeded
- `failed`: replay hit the current attempt ceiling or a permanent-looking failure path

The Settings screen intentionally hides `synced` entries and focuses on non-synced queue state.

## Replay Lifecycle

### 1. Queue creation

Offline-capable frontend code calls:

- `queueOfflineMutation(mutationType, mutationArgs)`

That writes a local IndexedDB entry with:

- JSON-serialized args
- `pending` status
- attempt count `0`

### 2. Replay triggers

Replay currently runs from the client in these situations:

- authenticated app startup
- browser `online` transitions
- manual `Process Queue` action from Settings

Background Sync may also be requested when supported by the browser, but replay cannot depend on it.

Operational rule:

- Background Sync is best-effort only, not required behavior.

### 3. Replay processing

`processOfflineQueue()`:

- loads pending entries
- marks each item `syncing`
- increments `attempts`
- parses and validates `mutationArgs`
- resolves a replay handler from the handler registry
- executes the real replay handler
- marks the item `synced` on success
- records the last successful local replay timestamp for diagnostics

### 4. Failure handling

If replay fails:

- unsupported mutation types stay visible with an explicit error
- corrupt payloads fail without poisoning the rest of the queue
- items return to `pending` until the max attempt threshold is reached
- items become `failed` once attempts reach `MAX_OFFLINE_REPLAY_ATTEMPTS`

Current retry model:

- best-effort only
- no exponential backoff
- no server-coordinated retry policy

## Replay Handler Model

Replay is opt-in per mutation type.

That is a deliberate constraint, not an accident.

Why:

- it prevents generic blind replay of arbitrary payloads
- it forces each offline-capable mutation family to define its own safety assumptions
- it keeps unsupported mutation types explicit instead of silently pretending they synced

Current live replay coverage:

- `userSettings.update`

That path currently covers queued changes originating from:

- theme preference updates
- timezone preference updates
- dashboard layout preference updates

## What Settings Means

The Offline settings surface is diagnostic, not authoritative server sync reporting.

The screen currently reports:

- browser online/offline state
- local IndexedDB queue counts
- per-item local replay state
- manual retry/remove controls for failed entries
- manual queue processing
- last successful local replay time
- service-worker and Background Sync capability signals

It does not mean:

- the server has accepted every recent user edit
- all product mutation families are replayable offline
- Background Sync is guaranteed

## Background Sync Policy

Current policy:

- use Background Sync when the browser exposes it
- never require it for correctness
- always keep foreground replay paths available

Practical fallback when Background Sync is unavailable:

- replay on authenticated app startup
- replay on reconnect
- replay from the Settings button

## Known Gaps

The current architecture still lacks:

- replay coverage for more mutation families
- explicit conflict-resolution policy
- idempotency guarantees for every future replayable mutation
- durable server-side auditing for failed offline actions
- end-to-end browser verification for installability, offline navigation fallback, and push across worker updates
- a single unambiguous worker ownership path at build time

## Extension Rules

When adding another offline-capable mutation family:

1. Add a specific queue producer for that mutation family.
2. Register an explicit replay handler.
3. Decide what makes a failure retryable versus effectively permanent.
4. Add unit coverage for queueing and replay behavior.
5. Update the Settings copy only if the capability is truly live.

Do not:

- add a generic catch-all replay path
- claim offline support for a feature without a real replay handler
- reintroduce a second queue model unless there is a concrete product or compliance need

## Verification Checklist

For local verification of the current architecture:

```bash
pnpm test --run src/lib/offline.test.ts src/hooks/useOffline.test.ts src/components/Settings/OfflineTab.test.tsx
pnpm build
```

For runtime inspection:

1. Build and preview the app in production mode.
2. Confirm `/service-worker.js` is the registered worker in DevTools.
3. Confirm queued `userSettings.update` entries appear in Settings while offline.
4. Reconnect or use `Process Queue` and verify the queue clears.
5. Check `Last Successful Replay` in Settings updates afterward.
